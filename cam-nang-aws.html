Cẩm nang AWS: Nền tảng, dịch vụ cốt lõi & vận hành hệ thống

Toàn bộ kiến thức để thiết kế, triển khai và vận hành hạ tầng trên AWS — từ IAM, VPC, EC2, S3 tới serverless, container, bảo mật, giám sát và tối ưu chi phí — kèm sơ đồ kiến trúc mẫu (chương 15) và cheatsheet lệnh (chương 17). Dùng ô tìm kiếm bên trái để tra nhanh bất kỳ dịch vụ, khái niệm hay lệnh nào.

Bắt đầu trong 60 phút — 5 việc làm ngay

  1. Tạo tài khoản & khoá root: bật MFA cho tài khoản root, không dùng root cho việc hằng ngày — tạo user quản trị qua IAM Identity Center (chương 03).
  2. Đặt hàng rào chi phí trước tiên: bật AWS Budgets với cảnh báo (ví dụ 5–10 USD) trước khi tạo bất kỳ tài nguyên nào (chương 13).
  3. Cài AWS CLI v2 và chạy aws configure, kiểm tra bằng aws sts get-caller-identity (chương 17).
  4. Chọn một Region cố định để thực hành — với Việt Nam thường là ap-southeast-1 (Singapore) — và làm mọi lab trong đó (chương 02).
  5. Dựng lab đầu tiên: 1 VPC chuẩn 2 AZ + 1 EC2 + 1 S3 bucket, rồi xoá sạch sau khi xong (chương 04–06).

Cách dùng cẩm nang

Đọc tuần tự chương 02 → 06 để có nền; các chương sau tra cứu theo nhu cầu. Quy ước trình bày: bảng = so sánh nhanh để ra quyết định, hộp vàng = lưu ý dễ mất tiền hoặc mất an toàn, khối code nền tối = lệnh/cấu hình chạy được. Ô tìm kiếm bên trái quét toàn bộ nội dung, không phân biệt dấu tiếng Việt.

Mô hình trách nhiệm chia sẻ (Shared Responsibility)

AWS chịu trách nhiệm bảo mật của cloud; bạn chịu trách nhiệm bảo mật trong cloud:

03Danh tính & phân quyền

IAM — trái tim của mọi request

Mọi lời gọi API tới AWS đều đi qua IAM để trả lời: ai đang gọi, và họ có được phép làm việc này trên tài nguyên này không? Làm chủ IAM trước khi làm chủ bất kỳ dịch vụ nào khác.

Giải phẫu một policy JSON

{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "AllowReadAppBucket",
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::my-app-assets",
      "arn:aws:s3:::my-app-assets/*"
    ],
    "Condition": { "IpAddress": { "aws:SourceIp": "203.0.113.0/24" } }
  }]
}
  • Effect: Allow hoặc Deny. Action: API được phép, dạng service:Operation, hỗ trợ wildcard (s3:Get*).
  • Resource: ARN áp dụng — chú ý S3 cần cả ARN bucket (cho ListBucket) lẫn /* (cho object).
  • Condition: điều kiện theo IP, tag, MFA, Org ID (aws:PrincipalOrgID), TLS (aws:SecureTransport)...

Logic đánh giá quyền

  1. Có explicit Deny ở bất kỳ đâu → từ chối, không cần xét tiếp.
  2. Không có Deny, có ít nhất một Allow áp dụng → cho phép.
  3. Không có gì → mặc định từ chối (implicit deny).

Khi có SCP, permission boundary, session policy: quyền hiệu lực là giao (intersection) của tất cả các lớp — một lớp không cho là không được.

Role & STS — danh tính tạm thời

STS (Security Token Service) phát credential tạm (AccessKey + Secret + SessionToken, có hạn dùng) khi AssumeRole. Các tình huống kinh điển:

Tình huốngCách chuẩn
EC2 gọi AWS APIGắn instance profile (role cho EC2) — SDK/CLI tự lấy credential qua metadata, không bao giờ chép access key vào máy.
Lambda / ECS taskExecution role (Lambda) / task role (ECS) — mỗi hàm/task một role tối thiểu quyền.
Cross-accountAccount B tạo role có trust policy tin account A; A AssumeRole sang B.
CI/CD (GitHub Actions...)OIDC federation: pipeline assume role bằng token OIDC — không lưu access key trong secrets (mẫu ở chương 12).
Kubernetes trên EKSIRSA hoặc EKS Pod Identity: từng pod một role riêng (chương 09).
Người đăng nhậpIAM Identity Center: SSO + permission set, credential ngắn hạn qua aws configure sso.

Chuẩn thực hành

  • Bật MFA cho root và mọi user quản trị; không bao giờ tạo access key cho root.
  • Least privilege: bắt đầu hẹp, mở dần; dùng IAM Access Analyzer để tìm quyền thừa và tài nguyên bị chia sẻ ra ngoài.
  • Ưu tiên role + credential ngắn hạn; nếu buộc dùng access key dài hạn thì xoay vòng định kỳ (~90 ngày).
  • Chặn theo ngữ cảnh bằng Condition: aws:SourceIp, aws:PrincipalOrgID, bắt buộc aws:MultiFactorAuthPresent cho hành động nhạy cảm.
  • Ở quy mô lớn, cân nhắc ABAC (phân quyền theo tag) thay vì nhân bản policy theo từng team.
  • Bật CloudTrail ngay từ đầu để mọi lời gọi API đều có vết (chương 10).
Sự cố phổ biến số 1: lộ access key khi push code lên GitHub. Phòng bằng cách dùng role/OIDC thay cho key, và quét repo bằng công cụ như Gitleaks / git-secrets trước khi commit.
04Mạng: VPC · DNS · CDN

VPC — nền móng mọi hệ thống trên AWS

Gần như mọi tài nguyên "có IP" (EC2, RDS, Lambda trong VPC, ECS...) đều nằm trong một VPC (Virtual Private Cloud) — mạng riêng ảo do bạn định nghĩa trong một Region. Hiểu chắc VPC là điều kiện tiên quyết trước khi động vào compute.

Kiến trúc VPC & subnet

  • CIDR của VPC: từ /16 (65.536 IP) đến /28 (16 IP). Chọn dải private (10.x, 172.16–31, 192.168) và không trùng với mạng on-premises hay VPC khác sẽ nối về sau.
  • Subnet thuộc một AZ duy nhất. Trong mỗi subnet, AWS giữ lại 5 địa chỉ IP (địa chỉ mạng, router, DNS, dự phòng, broadcast) — subnet /24 chỉ còn 251 IP dùng được.
  • Subnet public = có route 0.0.0.0/0 → Internet Gateway. Subnet private = không có route đó (ra Internet phải qua NAT).
  • Mẫu chuẩn production: mỗi AZ có 3 lớp subnet — public (ALB, NAT), private-app (EC2/ECS), private-db (RDS) — trải trên ít nhất 2 AZ.

Security Group vs Network ACL

Hai lớp "tường lửa" hay bị nhầm nhất — và là câu hỏi phỏng vấn kinh điển:

Tiêu chíSecurity Group (SG)Network ACL (NACL)
Gắn vàoENI / instance / service (mức tài nguyên)Subnet (mức mạng)

Loại ruleChỉ AllowAllow và Deny
Thứ tự xétCộng gộp tất cả ruleTheo số thứ tự, khớp rule đầu tiên là dừng

Route 53 · CloudFront · ELB — lớp tiếp nhận traffic

Route 53 — DNS managed (SLA 100%):

  • Hosted zone public (tên miền Internet) hoặc private (phân giải nội bộ trong VPC).
  • Alias record ưu việt hơn CNAME: trỏ thẳng tới tài nguyên AWS (ALB, CloudFront, S3 website...), dùng được ở zone apex (example.com trần) và miễn phí truy vấn.
  • Routing policies: Simple · Weighted (chia % — canary) · Latency (region gần nhất) · Failover (kết hợp health check làm DR) · Geolocation (theo nước) · Geoproximity · Multivalue (trả nhiều IP khỏe) · IP-based.
  • Health check giám sát endpoint từ bên ngoài, có thể gắn CloudWatch alarm và dùng làm điều kiện failover.

CloudFront — CDN với hàng trăm điểm biên (edge location):

  • Origin là S3 (khóa truy cập trực tiếp bằng OAC — Origin Access Control) hoặc bất kỳ HTTP origin nào (ALB, API GW).
  • Cache behavior theo path pattern (vd /static/* cache lâu, /api/* không cache); kiểm soát bằng TTL + cache policy; xả cache bằng invalidation.
  • HTTPS với chứng chỉ ACM — chứng chỉ cho CloudFront bắt buộc tạo ở us-east-1.
  • Chạy code tại biên: CloudFront Functions (JS siêu nhẹ, sửa header/redirect) và Lambda@Edge (logic nặng hơn).
  • Phát nội dung giới hạn bằng signed URL / signed cookie.
  • Anh em cùng nhà: Global Accelerator — không cache, cho traffic vào mạng xương sống AWS qua 2 IP anycast tĩnh; hợp với TCP/UDP, game, IP whitelist.

Auto Scaling Group — co giãn và tự chữa lành

  • Launch Template định nghĩa "máy trông thế nào" (AMI, type, SG, user data, IAM role) — có version, thay thế hoàn toàn Launch Configuration cũ.
  • ASG giữ số instance trong khoảng min ≤ desired ≤ max, rải đều nhiều AZ, tự thay máy hỏng.
  • Health check: mặc định theo EC2 status; bật thêm ELB health check để "app chết nhưng OS sống" cũng bị thay.
  • Chính sách scale:
    • Target tracking — giữ một chỉ số quanh mục tiêu (vd CPU 50%, request/target 1000). Đơn giản, nên dùng đầu tiên.
    • Step scaling — cộng/trừ theo bậc alarm.
    • Scheduled — theo lịch (giờ hành chính, sự kiện).
    • Predictive — ML dự báo theo chu kỳ ngày/tuần.
  • Warm pool giữ sẵn instance đã boot (stopped) để scale-out nhanh; lifecycle hook chèn bước chờ khi máy vào/ra (cài đặt thêm, rút log trước khi tắt).
  • Instance refresh — cuốn chiếu thay máy khi đổi Launch Template (rolling update AMI mới).
  • Gắn ASG vào target group của ALB: máy mới tự đăng ký, máy bị thu hồi tự rút khỏi LB.

S3 là trung tâm của gần như mọi kiến trúc

S3 chuyên sâu

  • Lưu object (file + metadata) trong bucket; tên bucket duy nhất toàn cầu, còn dữ liệu nằm trong Region bạn chọn.
  • Object tối đa 5 TB; upload trên 5 GB bắt buộc multipart upload (nên dùng từ ~100 MB — CLI tự làm).
  • Độ bền thiết kế 11 số 9 (99,999999999%); đọc-sau-ghi strong consistency.
  • Mặc định mọi object được mã hóa server-side (SSE-S3); nâng cấp lên SSE-KMS khi cần kiểm soát khóa + audit (chương 10).
  • Dung lượng không giới hạn, trả tiền theo GB lưu + request + data transfer ra.

Tính năng phải biết:

  • Versioning — giữ mọi phiên bản, cứu khỏi ghi đè/xóa nhầm; xóa chỉ tạo delete marker. Kết hợp MFA Delete cho bucket tối quan trọng.

  • Replication CRR/SRR (khác/cùng Region) — cần bật versioning hai đầu; phục vụ DR, tuân thủ, gom log.
  • Event notification → EventBridge/SQS/SNS/Lambda: "có file mới thì xử lý" — xương sống của pipeline hướng sự kiện.
  • Presigned URL — chia sẻ quyền tải/upload có thời hạn: aws s3 presign s3://BUCKET/KEY --expires-in 3600.
  • Static website hosting — host web tĩnh; production nên đứng sau CloudFront + OAC và khóa bucket private.
  • Object Lock (WORM) — chống xóa/sửa trong thời hạn (compliance, chống ransomware); Access Point — nhiều "cửa" policy riêng cho một bucket lớn.

  • Chạy hàm theo sự kiện, scale theo số request đồng thời, trả tiền theo GB-giây + số request. Cấp RAM 128 MB – 10 GB (CPU tỉ lệ theo RAM — hàm chậm đôi khi rẻ đi khi tăng RAM).
  • Cold start: lần gọi đầu phải khởi tạo môi trường. Giảm bằng: gói nhẹ, khởi tạo ngoài handler, provisioned concurrency (giữ sẵn môi trường ấm, tính phí), hoặc SnapStart (khôi phục từ snapshot — Java, và mở rộng cho Python/.NET từ 2024).
  • Version & alias: publish version bất biến, alias (vd prod) trỏ vào version và chia % traffic giữa 2 version → canary deploy (CodeDeploy tự động hóa được).
  • Layers — đóng gói thư viện dùng chung; container image tới 10 GB thay cho zip.
  • Lambda trong VPC: gắn ENI vào subnet private — muốn ra Internet phải có NAT; gọi dịch vụ AWS nên đi VPC endpoint.
  • Xử lý lỗi async: retry 2 lần → DLQ hoặc Destinations (đích riêng cho thành công/thất bại — nên dùng hơn DLQ).
  • Event source mapping — Lambda tự poll SQS/Kinesis/DynamoDB Streams theo batch; chỉnh batch size, batch window, và ReportBatchItemFailures để chỉ retry item hỏng.
  • Function URL — HTTPS endpoint trực tiếp không cần API GW (auth IAM hoặc public); hỗ trợ response streaming.

SQS · SNS · EventBridge — tách rời hệ thống

SQS — hàng đợi kéo (pull), bộ giảm chấn giữa các service:

  • Standard: gần như vô hạn throughput, at-least-once, có thể sai thứ tự → consumer phải idempotent.
  • Visibility timeout: message "tàng hình" khi đang được xử lý; xử lý xong phải xóa, quá hạn thì message hiện lại (retry tự nhiên).
  • Long polling (WaitTimeSeconds tới 20s) — giảm request rỗng, luôn bật.
  • DLQ + maxReceiveCount: message hỏng N lần thì sang hàng đợi chết để mổ xẻ; có nút redrive đẩy lại sau khi sửa lỗi.
  • FIFO (đuôi .fifo): đúng thứ tự + exactly-once trong 5 phút (dedup); thứ tự tính theo MessageGroupId; ~300 msg/s (3.000 với batch), cao hơn nữa ở chế độ high-throughput.
  • Message tối đa 256 KB; retention tới 14 ngày; delay queue/message timer hoãn phát tới 15 phút.

SNS — pub/sub đẩy (push): 1 publish → fan-out tới nhiều subscriber (SQS, Lambda, HTTP, email, SMS). Pattern chuẩn: SNS → nhiều SQS (mỗi consumer một queue, bền, retry riêng); filter policy để mỗi subscriber chỉ nhận message liên quan; có FIFO topic ghép FIFO queue.

EventBridge — event bus định tuyến theo nội dung sự kiện (pattern JSON):

  • Bus: default (sự kiện AWS: "EC2 stopped", "CodePipeline failed"...), custom (sự kiện nghiệp vụ của bạn), partner (SaaS: Datadog, Stripe...).
  • Rule khớp pattern → tới 5 target/rule (Lambda, Step Functions, SQS, API destination gọi webhook ngoài...).
  • Scheduler — cron/rate managed (thay cho CloudWatch Events cũ); Pipes — nối point-to-point nguồn→đích kèm filter/enrich; archive & replay — phát lại sự kiện cũ khi consumer từng lỗi; schema registry.

Mã hóa, bí mật, và bộ máy phát hiện

KMS · Secrets Manager · ACM · WAF

KMS — dịch vụ quản lý khóa, trung tâm của mọi mã hóa:

  • Khóa symmetric là mặc định; khóa không bao giờ rời KMS — mọi thao tác qua API (có vết CloudTrail).
  • AWS managed key (aws/s3, aws/ebs... — miễn phí, ít kiểm soát) vs customer managed key (phí theo tháng + request; tự đặt key policy, bật/tắt, dùng cross-account).
  • Mã hóa trực tiếp tối đa 4 KB; dữ liệu lớn dùng envelope encryption: GenerateDataKey → mã hóa cục bộ bằng data key → lưu data key đã mã hóa cạnh dữ liệu. Các dịch vụ AWS đều làm sẵn kiểu này.
  • Quyền = IAM policy giao với key policy (+ grants cho ủy quyền tạm). Bật automatic rotation mỗi năm cho CMK.
  • Multi-Region keys — cùng key ID ở nhiều Region (DR, Global Tables). Xóa khóa phải chờ 7–30 ngày (cửa hối hận) — mất khóa là mất dữ liệu vĩnh viễn.

11Giám sát & vận hành

CloudWatch, X-Ray và Systems Manager

CloudWatch — metrics, alarms, logs

  • Metrics: dịch vụ AWS tự phát (EC2 mặc định điểm 5 phút; bật detailed monitoring để có 1 phút). EC2 không có sẵn RAM/disk usage — cài CloudWatch Agent. Tự đẩy metric bằng PutMetricData hoặc Embedded Metric Format (ghi log là ra metric — rẻ và tiện cho Lambda).
  • Alarms: ngưỡng trên metric — "M trên N datapoints" để tránh nhấp nháy; action gọi SNS / Auto Scaling / EC2 stop-recover; composite alarm gộp nhiều alarm giảm nhiễu; nhớ xử lý trạng thái INSUFFICIENT_DATA (thiếu dữ liệu đôi khi chính là sự cố).
  • Logs: log group / log stream; retention mặc định là vĩnh viễn — đặt lại ngay (30–90 ngày cho app log) kẻo hóa đơn phình. Metric filter biến pattern log thành metric để đặt alarm (vd đếm "ERROR"); subscription filter stream log sang Lambda/Firehose/OpenSearch; Live Tail xem log realtime.

Truy vấn Logs Insights — câu lệnh dùng hằng ngày:

fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 20
  • Dashboards — ghép metric/log/alarm nhiều Region một màn hình; Synthetics canary — script "giả người dùng" gọi endpoint 24/7; RUM — đo trải nghiệm người dùng thật; Application Signals — APM tự động (SLO, service map) cho ứng dụng có instrument.
  • X-Ray — distributed tracing: nhìn một request xuyên qua API GW → Lambda → DynamoDB, tìm điểm chậm/lỗi; hướng chuẩn mở dùng ADOT (AWS Distro for OpenTelemetry).

Template YAML/JSON mô tả tài nguyên → CFN tạo/sửa/xóa cả stack theo đúng thứ tự phụ thuộc, lỗi thì tự rollback. Ví dụ tối giản:

  • State là trái tim: lưu S3 backend + khóa chống chạy đôi bằng DynamoDB, hoặc S3 native locking (Terraform ≥ 1.10, khỏi cần bảng DynamoDB).

14HA · DR · Well-Architected

Thiết kế cho thất bại: "everything fails, all the time"

Hai con số điều khiển mọi quyết định

  • RTO (Recovery Time Objective) — chịu ngừng dịch vụ tối đa bao lâu.
  • RPO (Recovery Point Objective) — chịu mất dữ liệu tối đa bao nhiêu (tính bằng thời gian). Hỏi nghiệp vụ hai số này trước, rồi mới chọn kiến trúc — số càng nhỏ, tiền càng lớn.

Khuôn mẫu High Availability trong một Region

  • Multi-AZ mọi tầng: ALB nhiều AZ, ASG rải AZ, RDS Multi-AZ, NAT mỗi AZ. Một AZ chết — hệ thống tự co lại chứ không tắt.
  • Stateless hóa app: session vào ElastiCache/DynamoDB, file vào S3/EFS → máy nào chết cũng thay được.
  • Health check + tự chữa lành: ELB đá máy hỏng, ASG thay máy mới — không ai phải dậy lúc 3 giờ sáng vì một instance.
  • Chống lỗi dây chuyền: retry với exponential backoff + jitter, timeout hợp lý, hàng đợi SQS làm giảm chấn, thao tác idempotent.
  • Giới hạn blast radius: tách account/môi trường, tách cell, canary deploy.

15Kiến trúc mẫu

Bốn bản thiết kế gặp đi gặp lại

1 · Web 3 lớp truyền thống (chuẩn production)

  • Scale: ALB + ASG target tracking; đọc DB nặng → thêm read replica / cache-aside với ElastiCache.
  • Bảo mật: SG dây chuyền (ALB → app → DB), secret trong Secrets Manager, vào máy bằng SSM.
  • Đây chính là kiến trúc để luyện tay đầu tiên — dựng được nó bằng Terraform/CDK là qua vòng gửi xe hầu hết JD.

2 · API serverless

  • Ưu: scale tự động theo request, trả tiền theo dùng, vận hành tối thiểu — hợp startup/MVP/tải thất thường.
  • Nhược cần quản: cold start (provisioned concurrency cho path nóng), giới hạn 15 phút (việc dài → Step Functions/Fargate), quan sát khó hơn (bắt buộc structured log + tracing).
  • Nâng cấp quen thuộc: DynamoDB Streams → Lambda cho side effect; Step Functions cho quy trình nhiều bước.

3 · Landing zone đa account (nền của tổ chức)

  • Vì sao đa account: cô lập sự cố/bảo mật (blast radius), hóa đơn tách bạch, quota riêng, quyền theo môi trường sạch sẽ.
  • Triển khai nhanh bằng Control Tower; chuẩn hóa baseline account bằng StackSets/Terraform.

4 · Data pipeline hướng sự kiện

  • Luồng chuẩn: nguồn đổ file vào S3 (raw) → EventBridge bắt sự kiện → Step Functions điều phối → Glue làm sạch/chuyển Parquet vào S3 (curated) → phân tích bằng Athena, trực quan hóa QuickSight.
  • Realtime thay S3 đầu vào bằng Kinesis/MSK → Flink → S3/OpenSearch (chi tiết dịch vụ ở chương 16).
  • Nguyên tắc: dữ liệu bất biến theo lớp (raw → curated), catalog tập trung (Glue Data Catalog), phân quyền Lake Formation.
